# CoreMatrix Home And Office Deployment Guide

This guide covers the common case where CoreMatrix runs on a machine in a home
or office network and does not have a public IP.

Typical goals:

- operators access CoreMatrix from the same LAN
- `cmctl` can manage the installation from a laptop or from the server itself
- the deployment stays production-like in process layout and data durability
- IM integration can start with Telegram polling and later graduate to webhook
  mode if a public HTTPS entrypoint is added

## Deployment Modes

### 1. LAN-only HTTP

Use this when all operators are on the same network and no outside service
needs to call back into CoreMatrix.

Recommended when:

- you are still bringing the system up
- you only need browser access and `cmctl`
- Telegram will use polling mode

Typical values:

```env
CORE_MATRIX_PUBLIC_BASE_URL=http://10.0.0.116:3000
RAILS_FORCE_SSL=false
RAILS_ASSUME_SSL=false
```

Behavior:

- browsers on the LAN open `http://10.0.0.116:3000`
- `cmctl` on operator laptops also uses `http://10.0.0.116:3000`
- `cmctl` on the server itself can use either `http://127.0.0.1:3000` or the
  LAN address
- Telegram polling works
- Telegram webhook does not work, because Telegram cannot reach a private HTTP
  address

### 2. Private HTTPS On The LAN

Use this when you want HTTPS and a stable hostname inside the office or home
network, but still do not expose CoreMatrix publicly.

Recommended when:

- you control local DNS or mDNS naming
- you want a reverse proxy to own TLS and host routing
- operators access CoreMatrix from multiple devices on the LAN

Typical values:

```env
CORE_MATRIX_PUBLIC_BASE_URL=https://corematrix.home.arpa
RAILS_FORCE_SSL=true
RAILS_ASSUME_SSL=true
```

Behavior:

- the reverse proxy terminates TLS and forwards to CoreMatrix
- operators use `https://corematrix.home.arpa` or another internal DNS name
- `cmctl` can use the same HTTPS hostname if the client trusts the certificate
- Telegram polling still works
- Telegram webhook still does not work unless this hostname is reachable from
  the public internet

### 3. Public HTTPS Edge

Use this when an external platform must call back into CoreMatrix.

Recommended when:

- you want Telegram webhook mode
- you want remote access from outside the LAN
- you want public HTTPS links generated by CoreMatrix

How to expose it:

- a public domain and reverse proxy on a public host
- a public reverse proxy or VPS that forwards into the LAN
- an HTTPS tunnel such as Cloudflare Tunnel, Tailscale Funnel, or a similar
  ingress product

Typical values:

```env
CORE_MATRIX_PUBLIC_BASE_URL=https://core.example.com
RAILS_FORCE_SSL=true
RAILS_ASSUME_SSL=true
```

Behavior:

- CoreMatrix still runs in the home or office
- the public edge terminates HTTPS and forwards to the app
- Telegram webhook can use the public URL
- Telegram polling also remains valid

## Choosing `CORE_MATRIX_PUBLIC_BASE_URL`

`CORE_MATRIX_PUBLIC_BASE_URL` is the address CoreMatrix uses when it generates
links for browsers and external systems. It should be the most operator-facing
origin, not an internal container hostname.

Use these rules:

1. If operators only use the LAN IP and no proxy exists yet, set it to the LAN
   HTTP address.
2. If you have an internal reverse proxy and internal HTTPS hostname, set it to
   that HTTPS origin.
3. If you later add a public HTTPS edge, change it to the public HTTPS origin.

Avoid these mistakes:

- do not set it to `http://core_matrix:80`
- do not set it to a Docker service name
- do not point it at `127.0.0.1` unless all consumers run on the same host

Keep this separate from internal control-plane URLs used by other services:

- `fenix` and `nexus` should keep using `CORE_MATRIX_BASE_URL` for
  container-to-container or host-local API access
- `CORE_MATRIX_PUBLIC_BASE_URL` is for humans and outside systems

## Reverse Proxy Guidance

For a production-like deployment on a single private host:

1. Keep CoreMatrix in the standard split topology:
   - `db`
   - `migrator`
   - `app`
   - `jobs`
2. Prefer a reverse proxy in front of the `app` service once you want a stable
   hostname or HTTPS.
3. Let the reverse proxy own `80` and `443`.
4. Either:
   - publish CoreMatrix directly on a LAN port such as `3000:80`, or
   - bind CoreMatrix to loopback only, such as `127.0.0.1:8080:80`, and let
     the reverse proxy talk to it on the same host

Practical recommendation:

- early bootstrap: publish `3000:80`
- after you add a reverse proxy: switch to `127.0.0.1:8080:80`

This keeps the Rails app private once the proxy is in place and avoids exposing
two competing entrypoints.

## TLS Guidance

TLS is optional for LAN-only HTTP, but recommended once you move beyond initial
bootstrap.

Use:

- no TLS for short-lived bootstrap on a trusted LAN
- private HTTPS for stable office or household use
- public HTTPS for webhook integrations or remote access

When TLS terminates before Rails:

- set `RAILS_ASSUME_SSL=true`
- usually also set `RAILS_FORCE_SSL=true`

When CoreMatrix itself is served as plain HTTP inside the LAN:

- set `RAILS_ASSUME_SSL=false`
- set `RAILS_FORCE_SSL=false`

## `cmctl` In A Private Network

There are two normal operator patterns:

### Operator laptop on the same LAN

Use the LAN or internal-HTTPS origin:

```bash
cmctl init
# Base URL: http://10.0.0.116:3000
```

### SSH onto the CoreMatrix server

Use a host-local URL:

```bash
cmctl init
# Base URL: http://127.0.0.1:3000
```

This keeps the server-side CLI independent from LAN DNS and still talks to the
same installation.

## Telegram Decision Tree

### Telegram Polling

Use polling when:

- the deployment has no public IP
- you want the simplest home or office setup
- you are still validating the integration

Requirements:

- recurring scheduler and queue workers must be running
- no public HTTPS is needed

Recommended default for private deployments:

- start with polling first
- only add webhook later if you truly need external callback delivery

### Telegram Webhook

Use webhook when:

- you have a public HTTPS entrypoint
- you want Telegram to push updates into CoreMatrix

Requirements:

- public HTTPS URL reachable by Telegram
- `cmctl ingress telegram-webhook setup`
- stable reverse proxy or tunnel
- set the webhook `base URL` to the public HTTPS origin, not a LAN address or
  Docker service name

If the server stays in a private LAN, add one of:

- a public reverse proxy
- a public VPS forwarding into the LAN
- an HTTPS tunnel product

## Recommended Progression

For home and office installs, the least painful progression is:

1. Bring up CoreMatrix in the production-like split topology.
2. Bootstrap it over LAN HTTP.
3. Run Codex and Telegram polling over the LAN first.
4. Add internal HTTPS when you want a nicer steady-state operator setup.
5. Add a public HTTPS edge only when you need webhook integrations.

This keeps the early deployment simple without baking in the wrong public URL
assumptions too early.
